# Artifact Ontology Guide

This page explains how the DSR Framework classifies, bounds, documents, and reviews artifacts. It is a human-facing guide to `models/dsr-artifact-ontology.yaml`; the YAML model remains the controlled source for entity names, validation implications, and downstream schema alignment.

## 1. Purpose

The artifact ontology helps DSR authors and reviewers answer six recurring questions:

1. What is the artifact?
2. Is the claimed contribution about an artifact type, an artifact instance, a design-knowledge object, or an artifact network?
3. What is inside the focal artifact, and what belongs to the surrounding context?
4. What make plan, use plan, and capacity specification identify the artifact?
5. What evidence supports artifact quality, artifact utility, and knowledge contribution claims?
6. What must be documented so another researcher can review, adapt, reuse, or extend the artifact responsibly?

The guide is intended for project planning, artifact specification, artifact review, package readiness checks, and contribution positioning.

## 2. Governing position

For this framework, a DSR artifact is a designed object of inquiry that must be:

- connected to a practical problem or improvement goal;
- documented at an explicit abstraction level;
- bounded from its implementation context and artifact network;
- described by a design specification where applicable;
- linked to demonstration, evaluation, or review evidence; and
- positioned as a contribution to design knowledge, practice, or both.

The strict ontological core treats the canonical DSR artifact as an **artifact universal**: a reusable artifact kind or type identified by a design specification, make plan, use plan, and capacity specification. The pragmatic documentation extension allows prototypes, deployments, software builds, examples, templates, checklists, and records to be described as artifact instances or documentary artifacts when they are the concrete evidence through which the artifact is demonstrated, evaluated, or reused.

### Review implication

A reviewer should be able to tell whether the submission claims its contribution from:

- the reusable artifact kind or design specification;
- one concrete instance, prototype, or deployment;
- a method, model, taxonomy, ontology, framework, checklist, template, or schema;
- evidence generated by demonstration or evaluation; or
- a resulting design-knowledge abstraction such as a design principle, design theory, or projectable pattern.

If the artifact claim blurs these levels, the artifact record is not yet reviewable.

## 3. Minimum artifact description

Every DSR artifact record should provide at least the following information.

| Field | What it answers | Review concern |
|---|---|---|
| `artifact_id` | What artifact is being identified? | Stable reference and traceability. |
| `artifact_name` | What should humans call it? | Clear communication. |
| `artifact_type` | What kind of artifact is it? | Controlled vocabulary alignment. |
| `abstraction_level` | Is it a type, instance, knowledge object, or network? | Contribution scope and reuse. |
| `purpose` | What practical or research purpose does it serve? | Relevance. |
| `problem_alignment` | What problem, objective, or requirement does it address? | Problem-space transparency. |
| `design_specification_status` | Is the make/use/capacity logic documented? | Build transparency and reviewability. |
| `evaluation_status` | What evidence exists or is planned? | Evidence sufficiency. |
| `contribution_claim_ids` | Which knowledge claims depend on this artifact? | Contribution transparency. |
| `boundary_conditions` | Where does the claim apply or not apply? | Projectability and non-overclaiming. |
| `evidence_links` | What records support the artifact claim? | Auditability. |

For L2 reviewable conformance, the artifact does not need full empirical validation, but it must be sufficiently described for a reviewer to identify the artifact, classify it, inspect its boundary, and judge whether claimed evidence is appropriate to the claim.

## 4. Core distinctions

### 4.1 Artifact universal versus artifact instance

| Distinction | Use when | Do not use when | Required documentation |
|---|---|---|---|
| Artifact universal | The contribution is intended to be reusable, adaptable, projectable, or generalizable beyond one local implementation. | The record only describes one deployment and does not abstract it into a reusable type. | Design specification, make plan, use plan, capacity specification, intended use effect, boundary conditions. |
| Artifact instance | The record describes a concrete prototype, software build, deployment, example package, or implementation. | The submission wants to claim broad reusable knowledge without explaining the reusable artifact type. | Instance version, environment, use context, instantiates-artifact link, demonstration or evaluation records. |
| Design knowledge object | The contribution is a design principle, method, model, framework, taxonomy, ontology, pattern, or theory. | The submission merely contains an implementation without abstraction. | Knowledge goal, knowledge scope, associated artifacts, evidence basis, boundary conditions. |
| Artifact network | The focal artifact depends on platforms, data, workflows, policies, standards, roles, or neighboring systems. | The network is used as a vague synonym for the artifact. | Focal artifact, network boundary, dependencies, integration assumptions, contextual conditions. |

Artifact instances are often necessary evidence, but they are not automatically the knowledge contribution. When a project reports a working implementation, it should still state whether the contribution is the implementation itself, the reusable design specification, the method embodied by the implementation, the evaluation finding, or a design principle abstracted from use.

### 4.2 Make plan, use plan, and capacity specification

A reviewable artifact should make its design logic visible.

| Component | Function | Typical evidence |
|---|---|---|
| Make plan | Explains how an artifact instance can be constructed, configured, assembled, generated, or maintained. | Build steps, configuration rules, source files, schema, production logic, dependency list, build record. |
| Use plan | Explains who uses the artifact, under what conditions, and for what intended effect. | User roles, use scenario, workflow, operating conditions, inputs/outputs, interaction sequence. |
| Capacity specification | States the capacities, functions, affordances, or performance potentials the artifact is expected to possess. | Capability statements, acceptance criteria, tests, indicators, evaluation criteria. |

The make plan supports artifact-quality claims such as completeness, conformance, reliability, maintainability, or technical correctness. The use plan supports utility claims such as usefulness, usability, fit for purpose, decision support, organizational improvement, or learning value. The capacity specification connects both: it states what the artifact must be capable of doing if the make plan and use plan are sound.

### 4.3 Use effect versus usage effect

The ontology distinguishes two kinds of effect:

| Effect type | Meaning | Attribution rule |
|---|---|---|
| Use effect | The effect intended to follow when the use plan is executed under specified conditions. | May be attributed to the artifact if the use conditions and evidence support the claim. |
| Usage effect | The broader effect that emerges in actual practice, mediated by users, context, workflows, organizational behavior, and neighboring artifacts. | Must not be attributed to the artifact alone unless the evaluation design justifies that attribution. |

This distinction protects the framework from overclaiming. A template may improve documentation completeness in a demonstration case, but broader organizational changes may depend on training, governance, incentives, review culture, and repository infrastructure.

### 4.4 Artifact versus system

A system boundary is not the same as an artifact boundary. Systems may include natural, social, organizational, technical, and environmental elements. A DSR artifact may be embedded in a system, but the record must state the focal designed object.

Use the following boundary questions:

1. What did the research team intentionally design or refine?
2. Which parts are required components of the artifact?
3. Which parts are external dependencies or contextual conditions?
4. Which parts are evidence, examples, demonstrations, or implementation settings?
5. Which social or organizational elements are designed components rather than background context?
6. What is excluded from the artifact claim?

If the artifact is a socio-technical system, the record should distinguish the designed configuration from the wider organization or social field.

## 5. Artifact classes

The framework supports multiple artifact classes while preserving the ontology's boundary discipline.

| Artifact class | Use for | Boundary note |
|---|---|---|
| Construct | Concepts, terms, variables, categories, or representations. | Not a full model or method unless relationships or procedures are specified. |
| Model | Structured representations of entities, relationships, processes, variables, states, or mechanisms. | State whether the model is conceptual, operational, explanatory, evaluative, prescriptive, or computational. |
| Method | Procedural artifacts with steps, inputs, outputs, roles, sequencing, or decision rules. | Distinguish the reusable method from one workflow instance applying it. |
| Instantiation | Prototype, software build, implementation, deployment, or concrete example. | Link to the artifact universal or state that findings are instance-specific. |
| Design theory | Prescriptive and explanatory design knowledge for a class of problems. | Must include scope, constructs, principles, mechanisms, and boundary conditions. |
| Framework | Organizing structure of dimensions, components, relationships, or decision points. | State whether it is classificatory, procedural, explanatory, evaluative, or integrative. |
| Taxonomy | Controlled classification system. | Include definitions and classification rules. |
| Ontology | Formal or semi-formal entity and relationship model. | Stronger than a glossary because it constrains relationships and interpretation. |
| Guideline | Prescriptive or advisory action guidance. | Advisory unless adopted as a normative requirement in a specification or checklist. |
| Template | Reusable file or document pattern for producing recurring information items. | Artifact status depends on encoded work logic, not emptiness. |
| Checklist | Review or readiness artifact with checks and pass/fail logic. | Must include reviewable criteria and remediation guidance. |
| Schema | Machine-readable validation artifact. | Validates records; does not prove artifact utility by itself. |
| Software or code | Executable or interpretable implementation of a method, model, workflow, algorithm, or system. | Distinguish code from the design knowledge it embodies. |
| Algorithm | Specified rule-governed transformation from inputs to outputs. | May be an artifact without being packaged as an application. |
| Socio-technical system | Designed arrangement of technical components, roles, routines, governance, and practices. | Requires explicit focal-artifact and context boundaries. |
| Documentary artifact | Protocol, policy, record, report, guide, or other text/file object designed for research, governance, or reuse. | Must be intentionally designed and positioned as part of the contribution. |

## 6. Materiality classes

Materiality identifies the form in which the artifact is embodied.

| Materiality | Typical artifacts | Review focus |
|---|---|---|
| Conceptual only | Construct, model, taxonomy, ontology, design theory. | Definition quality, coherence, boundary clarity, traceability. |
| Documentary | Protocol, checklist, template, schema, record, guide. | Usability, completeness, versioning, validation logic. |
| Digital | Software, code, database schema, algorithm, digital workflow. | Build reproducibility, dependencies, tests, runtime assumptions, security. |
| Physical | Device, instrument, physical prototype. | Material properties, manufacturing, safety, use context. |
| Socio-technical | Workflow system, operating model, intervention package. | Role design, governance, adoption context, attribution limits. |
| Mixed | Framework with templates and code, method with software support, packaged intervention. | Clear separation of components and evidence. |

## 7. Boundary and traceability rules

A reviewable artifact record should make boundary decisions explicit.

### Required boundary statements

- **Included elements:** components that are part of the focal artifact.
- **Excluded elements:** components or contexts intentionally outside the artifact claim.
- **Contextual elements:** dependencies or environmental conditions needed for use but not part of the artifact.
- **Evidence elements:** files, records, demonstrations, evaluations, examples, or outputs used to support claims.
- **Ambiguous elements:** unresolved boundary issues that require review or future refinement.
- **Boundary rationale:** reason for each major inclusion or exclusion.

### Traceability chain

At minimum, each artifact should be traceable through this chain:

```text
problem or objective
  -> artifact requirement
  -> design specification
  -> make plan / use plan / capacity specification
  -> artifact instance or design knowledge object
  -> demonstration or evaluation evidence
  -> contribution claim
  -> boundary condition and reuse guidance
```

If any link is missing, the record should state whether the gap is intentional, not applicable, or not yet mapped.

## 8. Demonstration, evaluation, and evidence

The artifact ontology distinguishes demonstration from evaluation.

| Evidence type | Purpose | Typical question |
|---|---|---|
| Demonstration | Shows that an artifact can be instantiated, interpreted, applied, or used in a plausible scenario. | Can this artifact be used as intended in at least one meaningful context? |
| Formative evaluation | Assesses and improves the artifact during design or construction. | Does the artifact have the specified capacities or required form? |
| Summative evaluation | Assesses artifact quality, utility, or effects after design stabilization. | Does the artifact satisfy its use goals and support its contribution claims? |
| Review evidence | Supports acceptance, conformance, readiness, or governance decisions. | Is the artifact sufficiently documented and quality-controlled? |

A demonstration may be sufficient for early reviewability, but it should not be described as evidence of full utility, impact, generalization, or publication readiness unless the evidence actually supports those claims.

## 9. Contribution positioning

The artifact record should connect to a contribution claim. The following distinctions should be preserved.

| Claim dimension | Allowed positions | Documentation requirement |
|---|---|---|
| Knowledge goal | definitional, descriptive, explanatory, predictive, explanatory-predictive, prescriptive, methodological, evaluative, normative | State what kind of knowledge the artifact contributes. |
| Knowledge scope | idiographic, nomothetic, mixed, projectable-to-problem-class, analytical-generalization | State whether the claim is local, general, or projectable with stated conditions. |
| Contribution type | artifact invention, method extension, model refinement, taxonomy, ontology, design theory, evaluation evidence, implementation evidence, reporting guidance | Link the contribution type to artifact evidence. |
| Evidence strength | not yet mapped, demonstrated, formatively evaluated, summatively evaluated, replicated, reused | Do not overstate evidence maturity. |

The artifact ontology therefore supports contribution transparency: the artifact is not merely an object in the package; it is the thing through which a claim to design knowledge is made inspectable.

## 10. Review checklist for artifact ontology alignment

Use this checklist when reviewing an artifact specification or package.

| Check ID | Review question | Pass condition |
|---|---|---|
| AO-01 | Is the focal artifact named and assigned a controlled artifact type? | Artifact name and type are present and compatible with the vocabulary. |
| AO-02 | Is the abstraction level stated? | Record distinguishes artifact universal, instance, design knowledge object, or artifact network. |
| AO-03 | Is the artifact boundary explicit? | Included, excluded, contextual, evidence, and ambiguous elements are identified. |
| AO-04 | Is the design specification sufficient for the artifact type? | Make plan, use plan, and capacity specification are present or waived with rationale. |
| AO-05 | Are artifact-quality claims linked to make-plan or capacity evidence? | Quality claims cite tests, inspection, schema validation, build records, or equivalent evidence. |
| AO-06 | Are utility claims linked to use-plan or use-context evidence? | Utility claims cite demonstration, evaluation, user review, field use, or justified artificial evaluation. |
| AO-07 | Are contribution claims separated from artifact description? | The record states what knowledge contribution follows from the artifact and evidence. |
| AO-08 | Are use effects distinguished from usage effects? | Broader practice outcomes are not attributed solely to the artifact without justification. |
| AO-09 | Is the artifact network bounded where relevant? | Dependencies, neighboring artifacts, platforms, workflows, and context are identified. |
| AO-10 | Are limitations and non-goals stated? | Scope limits, non-goals, and open boundary questions are documented. |

## 11. Common failure modes

| Failure mode | Why it matters | Remediation |
|---|---|---|
| Treating a whole organization or system as the artifact without a focal boundary. | Reviewers cannot tell what was designed or evaluated. | Define the focal artifact and list the surrounding system as context or artifact network. |
| Reporting only a prototype while claiming reusable knowledge. | The instance does not automatically establish a projectable contribution. | Add an artifact-universal description, design specification, and boundary conditions. |
| Treating evaluation results as the artifact. | Evidence and artifact are different information objects. | Link evaluation evidence to the artifact and contribution claim. |
| Omitting use conditions. | Utility cannot be interpreted without context. | Add intended users, use plan, use context, and use-effect statements. |
| Confusing use effect with usage effect. | Broader outcomes may be driven by people, governance, or context. | State attribution limits and contextual mediators. |
| Using artifact-type labels loosely. | Vocabulary drift weakens schema validation and review consistency. | Use controlled vocabulary entries and boundary notes. |
| Claiming generalizability from one local deployment. | Overclaiming reduces scholarly credibility. | Use projectability or analytical generalization with explicit boundary conditions. |

## 14. Assumptions and limitations

### Assumptions

- The framework treats the artifact universal as the strict core of a DSR artifact while allowing artifact instances and documentary artifacts to be represented as evidence-bearing outputs when this matches common DSR reporting practice.
- The guide assumes `models/dsr-artifact-ontology.yaml` and `vocabularies/artifact-types.yaml` will remain the controlled sources for schema and checklist alignment.
- The guide assumes downstream schemas and templates may tailor field requirements by artifact type and conformance level.

### Limitations

- This guide does not reproduce the full applied ontology or OntoUML formalism behind the source model.
- This guide does not settle difficult future cases such as adaptive artificial-intelligence systems, autonomous agents, non-technical institutional artifacts, laws, contracts, or maintenance-plan ontologies.
- This guide does not replace the evaluation framework; it only states what artifact claims need evidence.
- This guide is L2 reviewable, not L4 reusable or L5 archival-publication-ready.

## 15. Source basis summary

The artifact ontology is grounded primarily in the DSR artifact ontology model and its source basis. The central scholarly basis is Weigand, Johannesson, and Andersson's artifact ontology for DSR, especially its distinctions among artifact universals, artifact instances, make context, use context, design specification, design conversation, artifact networks, and use/usage effects. Contribution positioning is aligned with Akoka et al.'s knowledge-goal and knowledge-scope model. Transparency expectations are aligned with the six-feature DSR transparency framework, especially process, problem-space, solution-space, build, evaluation, and contribution transparency.
